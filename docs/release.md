# USTC Pay.gov Dev Server — Release & Publishing Playbook

This document describes how to prepare, version, publish, and validate releases for the **USTC Pay.gov Dev Server** (mock Pay.gov service used by USTC applications and the USTC Payment Portal).

Releases follow the **Changesets** workflow and publish an npm package consumed by internal systems.

Repository reference: **USTC Payment Test Server repo**  
https://github.com/ustaxcourt/ustc-pay-gov-test-server

---

## 1. Release Principles

- Every functional or behavioral change to the Dev Server **must** be tracked with a Changeset.
- Releases should be predictable, traceable, and validated with deployed integration tests.
- Artifacts (WSDL/XSD/HTML) required by the deployed service must be verified after each release when relevant.
- Compatibility with the **USTC Payment Portal repo** should be considered when changing SOAP/REST behavior.

---

## 2. When to Create a Release

Create a release when:

- API behavior changes (SOAP or REST).
- Static artifacts served by the service change (`html/pay.html`, `wsdl/*.wsdl`, `wsdl/*.xsd`).
- Authentication rules or environment variables change.
- A bug fix or enhancement is merged.
- Infrastructure adjustments affect runtime behavior.

If unsure, create a Changeset anyway — automatic release PRs allow maintainers to review version bumps.

---

## 3. Creating a Changeset

After developing your feature/fix on a branch:

```bash
npx changeset add
```

You will be prompted to select **patch**, **minor**, or **major**:

*   **Patch:** small fixes, logging improvements, internal-only adjustments
*   **Minor:** new features (e.g., simulated outcomes support)
*   **Major:** breaking behavior changes, changes to SOAP/REST fields, or new required environment variables

Changesets belong in the `./changeset/` directory.

Commit the Changeset file with your branch.

***

## 4. Submitting a Pull Request

Each PR that modifies server behavior should include:

*   [ ] A Changeset
*   [ ] Tests updated or added
*   [ ] Documentation updated (architecture, API, runbook, release notes if relevant)
*   [ ] No secrets committed
*   [ ] ADR added if architecture changed

Once merged into `main`, the automated **Release PR** will be generated by Changesets’ GitHub Action.

***

## 5. The Release PR (Auto‑Generated)

When a branch containing a Changeset is merged:

1.  A **Release PR** is automatically created.
2.  It contains:
    *   Updated `package.json` version
    *   Updated `CHANGELOG.md`
    *   Release metadata

**Do not edit version numbers manually.**  
Review the Release PR for correctness and merge when ready.

Merging the Release PR triggers:

*   npm publish
*   Git tag creation

***

## 6. Publishing (Automated)

Publishing is handled by GitHub Actions via the Release PR workflow.

When the Release PR is merged:

1.  The build is triggered.
2.  The package is published to npm under the internal namespace.
3.  `CHANGELOG.md` entries for the new version become part of the repo history.

You do not need to run `npm publish` manually.

***

## 7. Post‑Release Validation

After a release, **always validate the deployed dev instance**.

### 7.1 Update `.env.prod` locally

    BASE_URL=https://pay-gov-dev.ustaxcourt.gov
    ACCESS_TOKEN=<current token>

### 7.2 Run deployed integration tests

```bash
npm run test:integration:prod
```

These tests confirm:

*   The published version behaves correctly.
*   SOAP/REST endpoints respond.
*   Artifacts are correctly served.
*   Authentication is enforced properly.

If integration tests fail, perform a rollback (see below).

***

## 8. Post‑Deploy Artifact Verification

If your release impacts artifacts served by the Dev Server:

1.  Upload updated versions to the artifact S3 bucket:
    *   `html/pay.html`
    *   `wsdl/TCSOnlineService_3_1.wsdl`
    *   `wsdl/TCSOnlineService_3_1.xsd`
    *   `wsdl/tcs_common_types.xsd`
2.  Enable or verify S3 versioning for easy rollback.
3.  Fetch artifact URLs manually to ensure availability.

Example:

```bash
curl -I https://pay-gov-dev.ustaxcourt.gov/wsdl/TCSOnlineService_3_1.wsdl
```

Expect HTTP `200`.

***

## 9. Compatibility Testing With the Payment Portal

If changes affect SOAP/REST behavior, coordinate with maintainers of the:

**USTC Payment Portal repo**  
<https://github.com/ustaxcourt/ustc-payment-portal>

Validate:

*   Portal ↔ Dev Server interaction still aligns
*   Redirect flows still work (mock UI)
*   Token passing remains correct
*   No unexpected schema or field changes occurred

Update `/docs/compatibility.md` when necessary.

***

## 10. Rollback Procedure

If a release causes regressions:

### 10.1 npm rollback

Revert the version bump commit (Changesets automatically handles versioning on the next publish).

### 10.2 S3 artifact rollback

If artifacts were updated, restore prior versions using S3 versioning.

### 10.3 Token-related rollback

If authentication changes were made, restore prior token in your parameter store and notify consumers.

### 10.4 Re-run deployed integration tests

Confirm system health after rollback.

***

## 11. Manual Release Checklist

### Before publishing:

*   [ ] Changeset added
*   [ ] Tests passing locally
*   [ ] Integration tests passing locally
*   [ ] Docs updated (API, architecture, runbook)
*   [ ] Portal compatibility considered

### After publishing:

*   [ ] Deployed integration tests passing
*   [ ] S3 artifacts validated
*   [ ] Any downstream systems notified

***

## 12. Where to Ask Questions

    <https://github.com/ustaxcourt/ustc-pay-gov-test-server>

*   For Payment Portal compatibility: open an issue in the **USTC Payment Portal repo**  
    <https://github.com/ustaxcourt/ustc-payment-portal>

***

## 13. Summary

Releases for this server must be:

*   **Stable** — verified with deployed integration tests
*   **Traceable** — recorded via Changesets + changelog
*   **Coordinated** — validated with the Payment Portal
*   **Safe** — verify artifacts, authentication, and behavior

Following this playbook ensures predictable, safe updates to the mock Pay.gov Dev Server used across USTC systems.
